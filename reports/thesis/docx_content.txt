Abstract
Modern AI conversational systems suffer from a fundamental limitation: they lack persistent memory of the user. Each conversation begins from scratch, with no awareness of who the user is, what they have shared previously, how they feel, or how their beliefs and identity have evolved over time. This makes AI companions feel transactional and shallow rather than genuinely intelligent and empathetic.
Miryn-AI addresses this gap by building a context-aware AI companion backend with persistent memory, real-time emotion detection, named entity recognition (NER), identity tracking, and ML-powered memory ranking. The system is designed to remember users across sessions, detect shifts in their emotional state and personal beliefs, and intelligently rank which stored memories are most relevant to the current conversation.
The backend is implemented in Python using FastAPI and PostgreSQL with pgvector for semantic search. A dedicated Data Science (DS) service layer runs inference using HuggingFace Transformer models for emotion detection and spaCy for NER. An XGBoost-based memory ranking model is trained on 500 synthetic labelled examples and achieves an NDCG@5 of 0.99, Precision@1 of 0.62, and Recall@5 of 0.72. Analytics APIs expose emotion trends, identity volatility scores, semantic drift metrics, and version timelines. The system is fully containerized using Docker Compose and documented via Swagger/OpenAPI.
This report documents the complete architecture, design decisions, implementation details, data science pipeline, evaluation metrics, and future roadmap of the Miryn-AI system.

Chapter 1: Introduction
1.1 Motivation and Problem Statement
The rapid proliferation of large language models (LLMs) has made conversational AI accessible to millions of users worldwide. Systems like ChatGPT, Claude, and Gemini demonstrate remarkable language understanding and reasoning capabilities. However, a fundamental architectural limitation persists across virtually all commercially deployed AI chat systems: the absence of persistent user memory.
Each conversation session begins without any knowledge of prior interactions. The AI does not know the user's name, their recurring concerns, the emotions they expressed last Tuesday, or how their beliefs about themselves have changed over the past month. This creates an inherently transactional relationship between the user and the AI — one that cannot evolve into a genuine, context-aware companionship.
Consider a user who has been experiencing anxiety around their career for several months. Each time they begin a new session with a standard AI assistant, they must re-explain their situation from scratch. There is no continuity of care, no recognition of patterns, and no ability for the AI to proactively surface relevant memories or notice emotional deterioration over time.
Miryn-AI is built to solve this problem. It is a backend system for a context-aware AI companion that maintains persistent memory across sessions, detects and stores the user's emotional state in each message, extracts named entities (people, places, organisations), tracks the evolution of the user's identity and core beliefs over time, and uses a machine learning model to rank which memories are most relevant to retrieve for each new message.
1.2 Project Objectives
The primary objectives of this capstone project are:
Design and implement a persistent memory backend for an AI companion that stores messages, entities, and emotions across sessions.
Build a Data Science service layer capable of running real-time NER and emotion detection inference on every user message.
Implement an identity tracking system that captures the user's beliefs, values, and self-perception, and detects when these change over time.
Develop emotion analytics and identity analytics APIs that quantify mood trends, volatility, semantic drift, and identity stability.
Train a machine learning memory ranking model (XGBoost) that assigns relevance scores to stored memories based on five features: recency, emotional intensity, entity overlap, topic similarity, and identity alignment.
Evaluate the ranking model using Precision@K, Recall@K, and NDCG metrics and expose the ranked memory retrieval via a REST API endpoint.
Containerize the entire system using Docker Compose and document all endpoints via Swagger/OpenAPI.

Chapter 2: Background and Related Work
2.1 Persistent Memory in AI Systems
The challenge of giving AI systems persistent, long-term memory of users is an active area of research and product development. Early approaches relied on explicit user profiles stored in relational databases and retrieved using keyword matching. These systems suffered from rigid schema design and could not capture the nuanced, evolving nature of human identity and emotion.
More recent approaches leverage dense vector representations (embeddings) for semantic retrieval. Semantic memory systems encode past conversations as embedding vectors and retrieve the most contextually similar memories using cosine similarity search. The introduction of pgvector for PostgreSQL made it practical to perform approximate nearest neighbour (ANN) searches directly in the application database, eliminating the need for a separate vector store such as Pinecone or Weaviate.
MemGPT [Packer et al., 2023] proposed a hierarchical memory model for LLMs that distinguishes between in-context memory (the current conversation window) and external memory (long-term storage). This work demonstrated that intelligently paging memories in and out of the LLM context can significantly improve long-horizon task performance. Miryn-AI adopts a similar philosophy but extends it with explicit emotion and identity tracking layers.
2.2 Emotion Detection in Conversational AI
Sentiment analysis and emotion detection have a long history in NLP. Early lexicon-based approaches (e.g., VADER, SentiWordNet) assigned sentiment polarity to text based on word dictionaries. While fast, these approaches are brittle to sarcasm, context-dependence, and domain shift.
Transformer-based models pre-trained on large corpora and fine-tuned on emotion classification datasets have substantially improved accuracy. The model used in Miryn-AI — j-hartmann/emotion-english-distilroberta-base — is a DistilRoBERTa model fine-tuned on the GoEmotions dataset, capable of classifying text into seven emotion categories: anger, disgust, fear, joy, neutral, sadness, and surprise. Each classification also produces a continuous intensity score between 0 and 1, which Miryn-AI stores and uses as a feature in its memory ranking model.
2.3 Named Entity Recognition
Named Entity Recognition (NER) is the task of identifying and classifying named entities in text into predefined categories such as persons (PERSON), organisations (ORG), locations (GPE/LOC), dates (DATE), and events (EVENT). NER is a foundational component of information extraction pipelines.
spaCy is a widely-used industrial NLP library that provides a pre-trained English NER pipeline (en_core_web_sm) based on convolutional neural networks. In Miryn-AI, every user message is passed through spaCy's NER pipeline. Extracted entities are stored in the message metadata and in a dedicated entities table, allowing the system to build a rich knowledge graph of the people, places, and events that the user mentions over time.
2.4 Identity and Belief Tracking
The concept of tracking a user's 'identity' in AI systems is relatively novel. Prior work in personalised dialogue systems (e.g., Persona-Chat, BlenderBot) focused on modelling static user personas that do not evolve over time. Miryn-AI takes a dynamic approach: the identity model is updated as the user's expressed beliefs and self-descriptions change across sessions, and each update is recorded as a new 'version' in the identities table.
The semantic drift between successive identity versions is computed using cosine distance between the embedding vectors of each version. This provides a continuous measure of how much the user's self-perception has shifted, analogous to concept drift in machine learning but applied to human identity.
2.5 Learning to Rank for Information Retrieval
Learning to Rank (LTR) is a supervised ML approach to the problem of ordering items by relevance. Pointwise, pairwise, and listwise LTR methods have been extensively studied in the context of web search and document retrieval. XGBoost, a gradient-boosted tree ensemble method, has demonstrated strong performance on tabular ranking tasks due to its ability to model non-linear feature interactions and its robustness to outliers.
In the context of memory retrieval for AI companions, the ranking problem is to order a set of stored memories by their predicted relevance to the current user message. Miryn-AI frames this as a pointwise regression problem: the model predicts a relevance score between 0 and 1 for each (message, memory) pair, and memories are retrieved in descending score order.

3.2 Database Schema
The PostgreSQL database schema is designed around the following core tables:
3.2.1 Users and Authentication
The users table stores user credentials and profile information. Authentication is implemented using JSON Web Tokens (JWT). The get_current_user_id dependency in app/core/security.py validates the Bearer token on every protected endpoint and injects the authenticated user ID into the request context.
3.2.2 Messages
The messages table is the central table of the system. Each row represents a single message in a conversation. The schema includes a metadata JSONB column that stores the extracted emotions and entities for each message. This denormalised storage approach ensures that the raw extraction results are always available alongside the message, even if the downstream analytics tables are incomplete.
3.2.3 Memories
The memories table stores vector embeddings of user messages and important fragments of conversation. Each row contains a 384-dimensional embedding vector (computed by all-MiniLM-L6-v2) stored as a pgvector column. Semantic similarity search is performed using pgvector's cosine distance operator (<=>), enabling sub-millisecond retrieval of the top-K most contextually similar memories.
3.2.4 Entities
The entities table stores named entities extracted from messages. Each entity has a type (PERSON, ORG, GPE, DATE, etc.), a value, and a foreign key to the source message. Over time this table accumulates a rich knowledge graph of the people, places, and events in the user's life.
3.2.5 Identities
The identities table implements a versioned record of the user's identity. Each row represents one 'version' of the user's self-model at a point in time. The table stores the raw belief text, its embedding vector, and a timestamp. The identity analytics service computes drift between successive versions as the cosine distance between their embedding vectors.

3.3 Request Lifecycle
A typical user message follows this lifecycle through the system:
User sends a message via the frontend. The React client sends a POST /chat request with a Bearer JWT token.
The FastAPI backend authenticates the request via get_current_user_id and routes it to the chat orchestrator (app/services/orchestrator.py).
The orchestrator dispatches the message concurrently to the DS service and to the LLM (via asyncio.gather). The DS service performs NER and emotion detection in parallel with the LLM call, ensuring no latency penalty.
The extracted emotions and entities are stored in the messages table's metadata JSONB column.
The memory ranking model is queried to retrieve the top-K most relevant historical memories. These are injected into the LLM prompt context.
The LLM generates a response using the enriched context. The response is returned to the user and stored as a new message.
Background Celery tasks update the identity model and analytics tables asynchronously.

The backend application follows a layered architecture with clear separation between API routes, service logic, and data access. The directory structure of the backend is:

4.2 Authentication and Security
Authentication is implemented using industry-standard JWT (JSON Web Token) bearer tokens. The app/core/security.py module exposes the get_current_user_id FastAPI dependency, which is injected into every protected endpoint. The dependency validates the token signature, checks expiry, and extracts the user ID. Rate limiting is enforced via a custom RateLimitMiddleware that uses Redis to count requests per user per minute.
4.3 Chat Orchestrator
The orchestrator (app/services/orchestrator.py) is the central coordination module. It is responsible for:
Receiving a validated chat message and user context.
Spawning concurrent inference tasks using asyncio.gather — the DS service (NER + emotion detection) and the LLM completion call run in parallel.
Persisting the message and its metadata (extracted emotions, entities) to the database.
Retrieving relevant memories using the ranking model and injecting them into the LLM prompt.
Returning the LLM's response with full metadata including detected emotions and entities.
A key design decision in the orchestrator is to run DS inference off the main event loop using asyncio's thread pool executor. This prevents the CPU-bound HuggingFace model inference from blocking the async event loop and degrading API response times for other concurrent requests.

Chapter 5: Data Science Service Layer
5.1 Overview
The Data Science (DS) service layer is one of the most technically complex and novel components of Miryn-AI. It is implemented in app/services/ds_service.py and is responsible for running real-time ML inference on every user message. The service is designed to be non-blocking: all model inference runs off the asyncio event loop in a thread pool, ensuring it does not degrade API throughput.
The DS service loads three ML models at startup:
Emotion Detection Model: j-hartmann/emotion-english-distilroberta-base (HuggingFace Transformers)
Sentence Embedding Model: all-MiniLM-L6-v2 (SentenceTransformers, 384-dimensional output)
NER Pipeline: spaCy en_core_web_sm
Threading locks are used to protect model inference from concurrent access, as the HuggingFace models are not thread-safe by default. The HuggingFace cache path is configured via the HF_HOME environment variable to a stable location (/opt/huggingface) inside the container, ensuring models are pre-downloaded into the Docker image and not re-downloaded at runtime.

5.3 Named Entity Recognition
NER is performed using spaCy's en_core_web_sm pipeline. The pipeline identifies the following entity types in user messages:
PERSON — Names of people mentioned by the user (e.g., 'my sister Priya', 'Dr. Sharma')
ORG — Organisations (e.g., 'Google', 'DIT University')
GPE / LOC — Countries, cities, geographic locations (e.g., 'Delhi', 'Paris')
DATE / TIME — Temporal references (e.g., 'next Monday', 'last March')
EVENT — Named events (e.g., 'the interview', 'my exam')
Extracted entities are stored in both the messages.metadata JSONB column and the dedicated entities table. Over time, this builds a queryable knowledge graph of the user's world. For example, if a user mentions 'Google' in 15 separate messages, the system can recognise Google as a persistent, high-salience entity in that user's life.
5.4 Sentence Embeddings
The all-MiniLM-L6-v2 model from SentenceTransformers produces 384-dimensional dense vector embeddings for text inputs. These embeddings capture the semantic meaning of text in a continuous vector space: semantically similar texts have embeddings with high cosine similarity, while semantically dissimilar texts have low cosine similarity.
Embeddings are generated for:
Every user message — stored in the memories table for semantic retrieval.
Every identity version — stored in the identities table for semantic drift computation.
The pgvector extension stores these 384-dimensional vectors as native PostgreSQL vector columns, enabling efficient cosine similarity queries using the <=> operator. This allows the system to retrieve the top-K semantically most similar memories in sub-millisecond time, even at scale.
5.5 Concurrent Inference with asyncio.gather
A key engineering decision in the DS service integration is the use of asyncio.gather to run DS inference concurrently with the LLM completion call. Both operations are I/O-bound from the asyncio perspective (they block the thread, not the event loop), so running them concurrently via a thread pool executor provides significant latency savings.
The implementation uses asyncio.get_event_loop().run_in_executor(None, ...) to offload the CPU-bound model inference to the default thread pool, then awaits both the DS result and the LLM result together with asyncio.gather. In practice this reduces the total latency of a chat request by approximately the time of DS inference (typically 100–400ms depending on message length).
Chapter 6: Analytics APIs — Emotion, Identity and Memory
6.1 Overview of Analytics Architecture
The analytics layer of Miryn-AI is one of its most distinctive features. Rather than simply storing data, the system actively analyses patterns in the user's emotional state and identity over time, exposing quantitative metrics via REST APIs. These metrics are designed to be both psychologically meaningful (to end users and practitioners) and technically rigorous (to demonstrate ML and data engineering competence).
The analytics APIs are implemented in three files:
app/services/emotion_analytics.py — Emotion trend and volatility analysis
app/services/identity_analytics.py — Identity stability and semantic drift analysis
app/api/analytics.py — Route handlers exposing GET /analytics/emotions, /identity, and /summary
6.2 Emotion Analytics
The emotion analytics service analyses the sequence of emotion records stored in the user's message history over a configurable time window (default: 30 days). It computes the following metrics:
6.2.1 Mood Score
The mood score is a single scalar value between -1.0 (maximally negative) and +1.0 (maximally positive) that summarises the user's overall emotional state over the analysis window. It is computed as a weighted average of emotion intensities, with positive emotions (joy, surprise) assigned positive weights and negative emotions (sadness, anger, fear, disgust) assigned negative weights.
6.2.2 Emotional Volatility
Volatility measures how much the user's dominant emotion changes from message to message. High volatility indicates rapid emotional swings; low volatility indicates emotional stability. Technically, volatility is computed as the standard deviation of the mood score time series, normalised to [0, 1].
6.2.3 Emotional Trend
The trend indicates whether the user's mood is improving, stable, or deteriorating over the analysis window. It is computed as the slope of a linear regression fit to the mood score time series. A positive slope indicates an improving trend; a negative slope indicates deterioration.
6.2.4 Emotional Entropy
Entropy measures the diversity of the user's emotional range. A user who expresses only one emotion repeatedly (e.g., always neutral) has low entropy. A user who expresses a wide variety of emotions has high entropy. Entropy is computed using the Shannon entropy formula applied to the empirical distribution of emotion labels over the analysis window.
6.2.5 Dominant Emotions
The dominant emotions are the top-3 most frequently expressed emotions during the analysis window, ranked by their total accumulated intensity score.
Fig. 6.1: Mood Score Timeline and Emotional Volatility (30-day window)
6.3 Identity Analytics
Identity analytics operates on the versioned identity records stored in the identities table. Each time the system detects a significant shift in the user's expressed beliefs or self-description, a new identity version is created. The identity analytics service analyses the sequence of versions to compute:
6.3.1 Stability Score
The identity stability score is a value between 0 (completely unstable — identity changes every session) and 1 (perfectly stable — identity has never changed). It is computed as 1 minus the mean pairwise semantic drift between successive identity versions.
6.3.2 Semantic Drift
Semantic drift is the central metric of the identity analytics system. For each pair of consecutive identity versions (v_i, v_{i+1}), the drift is computed as the cosine distance between their embedding vectors:
drift(v_i, v_{i+1}) = 1 − cosine_similarity(embed(v_i), embed(v_{i+1}))
A drift value of 0 indicates no change between versions. A drift value of 1 indicates maximal semantic divergence. In practice, even significant identity shifts produce drift values in the range 0.1–0.5.
6.3.3 Index Volatility
Index volatility extends the drift metric to measure the variance of drift values across all version transitions. A user whose identity drifts consistently (always by approximately the same amount) has low index volatility. A user whose identity oscillates — sometimes stable, sometimes wildly shifting — has high index volatility. This metric can flag pathological patterns such as identity fragmentation.
6.3.4 Version Timeline
The version timeline is a chronological list of all identity versions with their timestamps and drift values. This allows the system (and the user) to identify specific moments when significant identity shifts occurred, correlate them with emotional events, and understand the trajectory of identity evolution over time.

Chapter 7: Memory Ranking ML Model
7.1 Problem Formulation
The memory ranking problem is formulated as a supervised pointwise regression task. Given a current user message and a set of stored memories, the model must assign a relevance score s ∈ [0, 1] to each (message, memory) pair. Memories are then retrieved in descending order of their predicted relevance score, and the top-K memories are injected into the LLM's context window.
This approach improves upon pure vector similarity search (which retrieves memories based solely on semantic text similarity) by incorporating additional signals: recency, emotional importance, entity overlap, and identity alignment. A memory about the user's Google interview may be semantically dissimilar to a message about job anxiety, yet highly relevant. The ML model is designed to capture these non-obvious relevance signals.

7.3 Training Data Generation
A key challenge in building the memory ranking model was the absence of labelled training data (since Miryn-AI had no real users at the time of model development). The solution was to generate synthetic training data using a principled algorithmic approach.
The data generation script (app/services/generate_training_data.py) creates 500 labelled (message, memory, relevance_score) triplets. The relevance score for each triplet is computed as a weighted combination of the four features:
score = 0.35 × text_overlap + 0.20 × recency + 0.20 × emotional_intensity + 0.15 × entity_overlap + 0.10 × identity_alignment + ε
where ε is a small Gaussian noise term added to prevent overfitting to the exact feature weights. The 500 examples cover 20 diverse message templates and 25 memory templates across domains including work, relationships, health, emotions, goals, and personal milestones.

8.2 Ranking Metrics
The primary evaluation framework for a memory ranking system is information retrieval metrics, which measure the quality of the ranked list rather than individual score accuracy. The evaluation script (app/services/evaluate_model.py) groups the 500 examples into batches of 10 (simulating a pool of 10 candidate memories per query) and computes the following metrics:
8.2.1 Precision@K
Precision@K measures what fraction of the top-K retrieved memories are actually relevant (relevance score ≥ 0.5). A high Precision@1 indicates the model reliably surfaces the single most relevant memory at the top of the ranked list.
8.2.2 Recall@K
Recall@K measures what fraction of all relevant memories are captured in the top-K retrieved set. A high Recall@5 indicates the model rarely misses relevant memories when retrieving the top 5.
8.2.3 NDCG@K
Normalised Discounted Cumulative Gain (NDCG) is the gold-standard ranking metric. It evaluates not just whether relevant memories are retrieved, but whether they are ranked in the correct order. NDCG penalises relevant memories that are ranked lower than they should be, using a logarithmic discount factor. An NDCG of 1.0 indicates a perfectly ordered ranked list.

Chapter 9: Use Cases and Application Scenarios
9.1 Mental Health Companion
The primary intended use case for Miryn-AI is as a mental health and emotional wellness companion. The system's persistent memory, emotion detection, and identity tracking capabilities are directly aligned with the needs of users seeking continuous emotional support.
In this use case, a user interacts with the AI companion over several weeks or months. The system tracks the user's emotional trajectory, detects patterns such as persistent sadness or increasing anxiety volatility, and proactively incorporates relevant past context ('Last week you mentioned feeling stressed about your interview — how did it go?') into the AI's responses. The identity analytics layer can detect when the user's self-perception shifts significantly, which may indicate important life events or psychological transitions.
9.2 Personalised Productivity Assistant
Miryn-AI can function as a personalised productivity and journaling assistant. Users can log their daily thoughts, goals, and progress. The NER pipeline extracts project names, deadlines, and people mentioned, building a structured knowledge graph of the user's professional life. The memory ranking model ensures that when the user asks 'What should I focus on today?', the most relevant recent memories (upcoming deadlines, unresolved tasks, important meetings) are surfaced in the AI's context.
9.3 Long-Term Relationship and Life Journal
Users can use Miryn-AI as a long-term life journal: a record of their experiences, relationships, emotions, and growth. The identity version timeline provides a quantitative narrative of how the user has changed over months or years. The semantic drift metric can identify inflection points — moments when the user's worldview or self-concept shifted significantly — that correspond to major life events.
9.4 Research and Clinical Applications
The emotion analytics and identity analytics APIs expose quantitative signals that could be valuable in research and clinical contexts. The mood score trajectory, emotional volatility, and identity drift metrics provide objective, longitudinal measurements of psychological state that could supplement traditional assessment tools. The API-first design makes integration with external research platforms straightforward.

Chapter 11: Conclusion and Future Work
11.1 Summary of Contributions
This capstone project successfully designed and implemented a production-grade backend for an AI companion with persistent memory, real-time emotion and entity extraction, identity tracking, and ML-powered memory ranking. The key contributions of this work are:
A complete FastAPI backend with 10 REST API endpoints, JWT authentication, rate limiting, and Swagger documentation.
A Data Science service layer that runs emotion detection (7-class DistilRoBERTa), NER (spaCy), and sentence embedding (all-MiniLM-L6-v2) inference concurrently with LLM calls, with no blocking of the async event loop.
An emotion analytics system computing mood score, volatility, trend, entropy, and dominant emotions from longitudinal message data.
An identity analytics system computing stability score, semantic drift, index volatility, and version timeline from versioned identity records.
A synthetic training data pipeline generating 500 labelled (message, memory, relevance_score) triplets for ML model training.
An XGBoost memory ranking model achieving RMSE 0.054, NDCG@5 0.99, and Recall@5 0.72.
A fully containerized Docker Compose deployment with 7 services including PostgreSQL, Redis, Celery, and the FastAPI backend.
11.2 Limitations
The current system has several limitations that should be acknowledged:
The memory ranking model is trained on synthetic data. Performance on real user data may differ and will need to be validated once the system has real users.
The identity model update logic currently relies on heuristic triggers rather than a learned classifier for detecting significant belief shifts.
The DS service loads models into memory at startup, which requires approximately 2GB of RAM. This limits deployment on resource-constrained infrastructure.
The emotion detection model was trained on English text and may underperform on code-switched or non-English input.
11.3 Future Work
Several directions for future development have been identified:
Re-train the memory ranking model on real user interaction data as the system accumulates users, replacing the synthetic training set with ground-truth relevance labels collected through implicit feedback.
Implement a learned identity shift detector that uses the DS service's NER and embedding outputs to automatically trigger identity version creation when semantic drift exceeds a learned threshold.
Add multi-modal memory support: allow users to store images, voice notes, and documents as memories, with corresponding embedding models for each modality.
Deploy the system to Render cloud platform for production availability, enabling real user testing and data collection.
Implement the User A vs User B comparative drift visualisation as an interactive frontend dashboard component.
Explore transformer-based neural ranking models (e.g., cross-encoders) as a replacement for XGBoost, potentially improving ranking quality at the cost of higher inference latency.
